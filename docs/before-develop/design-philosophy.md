# @echolab-auto/ui-frame 设计理念

> 本文档面向参与本项目的开发者。理解这些设计原则后，你应当能够在不查看现有组件源码的情况下，预判新组件该如何实现。

---

## 一、物理隐喻优先

本库不是"通用 UI + 阴影"，而是以新拟态设计语言为核心的系统性实现。所有视觉效果必须有物理对应：

- **凸起** = 物体浮于表面，产生底部阴影
- **凹陷** = 物体陷入表面，内部产生反向阴影
- **elevation** = 台阶高度，正数凸起、负数凹陷、零值平齐

设计决策从物理直觉出发，而非装饰性选择。缓动曲线也服务于物理隐喻：弹性过冲模拟材料回弹，快速压缩模拟按压干脆感。

---

## 二、Headless 与 UI 解耦

业务逻辑与视觉呈现彻底分离：

- **Headless composables** 封装纯逻辑（状态管理、键盘导航、分页计算等），零渲染、零样式
- **Vue 组件层** 只负责新拟态视觉表达和 DOM 结构

逻辑层可以被单独使用（自定义 UI），UI 层也可以通过替换组件实现来复用（自定义逻辑）。两者通过标准接口连接，互不依赖。

---

## 三、级联配置体系

所有可配置属性遵循严格的**三级级联**：

```
显式 prop > 全局配置 > 硬编码默认值
```

全局配置通过 Vue Provide 注入，任何组件都可以读取并按此优先级解析自身属性。使用者既能覆盖单个实例，也能统一调整全库行为，且无需修改源码。

---

## 四、Token 驱动的主题系统

所有视觉参数通过 CSS 自定义属性暴露，而非硬编码在 SCSS 中。这带来三个能力：

- **运行时主题切换**：亮色/暗色切换无需重新编译
- **外部可覆盖**：使用者通过修改 CSS 变量即可自定义外观
- **组件隔离**：视觉参数与实现解耦，主题变更不影响组件逻辑

暗色主题不是反色，而是重新调整阴影比例——暗阴影大幅增强、亮阴影极度减弱，以在低光背景下维持浮雕的可感知性。

---

## 五、协议化组件通信

复杂的组件组合关系不通过 prop drilling，而是通过**类型化的 Provide/Inject 协议**建立。Provider 定义通信契约，Consumer 按契约接入。这种设计让组件组合保持松散耦合，新增组合方式无需修改现有组件。

---

## 六、无障碍是基本要求

键盘导航、焦点管理、ARIA 属性、`prefers-reduced-motion` 支持、44px 触控目标——这些不是可选增强，而是组件合格的门槛。每个交互组件的实现都必须通过无障碍检查清单。
