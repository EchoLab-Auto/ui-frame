# Agent 开发验证 Pipeline

> 本 pipeline 适用于 AI Agent 参与 @echolab-auto/ui-frame 项目开发时的标准化验证流程。

---

## 前置检查（任何修改前必做）

```
1. 读取 docs/before-develop/design-philosophy.md
   → 确认理解六条核心设计原则

2. 确认修改范围类型
   → feat/fix/refactor/docs/style/test/build/ci/chore
```

---

## 开发阶段验证（编码中）

### Step 1 — 架构合规检查

| 检查项        | 规则                                                |
| ------------- | --------------------------------------------------- |
| Headless 分离 | 业务逻辑写在 composable，Vue 组件只做视觉表达       |
| 三级级联      | 显式 prop > 全局配置 > 硬编码默认值                 |
| Token 驱动    | 视觉参数用 CSS 变量，不硬编码在 SCSS                |
| 协议通信      | 复杂组合用类型化 Provide/Inject，不用 prop drilling |
| 物理隐喻      | 阴影、缓动必须有物理对应（凸起/凹陷/elevation）     |

### Step 2 — 代码质量即时检查

```bash
# 类型检查（每次保存后建议执行）
npm run typecheck

# Lint 检查（必须零错误才能提交）
npm run lint

# 自动修复（如有报错，先执行修复）
npm run lint -- --fix
```

> ⚠️ **重要**：`npm run lint` 出现任何 `prettier/prettier` 或 `eslint` 错误时，必须先修复再提交。不可将格式错误推送到 CI。
>
> 建议开启编辑器保存时自动格式化，避免手动修复：
>
> - VS Code：安装 Prettier + ESLint 插件，开启 `editor.formatOnSave` 和 `editor.codeActionsOnSave.source.fixAll.eslint`

### Step 3 — 测试驱动（如修改逻辑）

```bash
# 相关测试先行/随行
npm run test:watch    # 开发时监听模式
npm test              # 提交前全量验证
```

**测试要求**：

- 修改 composable → 必须更新/新增同名 `.test.ts`
- 修改组件 → 验证组件渲染和交互
- 覆盖率不能低于阈值（statements/lines/functions 80%, branches 70%）

## 人类闭环验证（UI 修改必做）

> Agent 对组件视觉、样式、交互做任何修改时，必须等待人类确认后方可进入后续流程。

### 触发条件

- 修改/新增 Vue 组件（`.vue`）
- 修改/新增样式文件（`.scss` / `.css`）
- 修改组件视觉相关的 composable 逻辑
- 调整主题 Token、动画、阴影等视觉参数

### 验证流程

```
┌────────────────────────────────────────────────────────────┐
│  Step 1 — Agent 启动本地预览                                │
│  ├─ 执行：npm run example                                    │
│  ├─ 示例站点启动后输出本地访问地址（如 http://localhost:5173）│
│  ├─ 确保修改在示例站点中有对应展示：                         │
│  │   ├─ 组件修改 → 在「组件总览」页可查看                    │
│  │   └─ 排版/布局修改 → 在「排版系统」页可查看               │
│  └─ Agent 告知人类预览地址及需要重点关注的区域               │
└────────────────────────────────────────────────────────────┘
                            ↓
┌────────────────────────────────────────────────────────────┐
│  Step 2 — 人类通过浏览器审查（审查人勾选）                   │
│  □ 视觉效果：是否符合物理隐喻（凸起/凹陷/elevation）        │
│  □ 交互行为：hover / focus / active 状态是否自然            │
│  □ 主题覆盖：亮色/暗色双主题是否正常                        │
│  □ 响应式：不同尺寸/断点下是否表现一致                      │
│  □ 排版系统：布局、栅格、容器在实际场景中是否正常            │
│  □ 无障碍：ARIA、焦点管理、prefers-reduced-motion           │
│  □ 一致性：与现有组件视觉语言是否统一                       │
└────────────────────────────────────────────────────────────┘
                            ↓
┌────────────────────────────────────────────────────────────┐
│  Step 3 — 反馈决策                                          │
│                                                             │
│  ✅ 通过 → 进入「提交前验证」                               │
│                                                             │
│  ❌ 不通过 → Agent 接收反馈重新修改                         │
│     ├─ 人类提供：具体问题描述 + 期望效果                     │
│     ├─ Agent 分析问题并重新编码                             │
│     ├─ 回到 Step 1 再次进入人类验证                         │
│     └─ 循环直至通过                                         │
└────────────────────────────────────────────────────────────┘
```

### 反馈模板

人类不通过时，建议按以下格式反馈：

```
【问题类型】视觉 / 交互 / 主题 / 响应式 / 无障碍 / 一致性
【具体问题】<详细描述>
【期望效果】<预期行为或参考样式>
【优先级】阻塞 / 建议优化
```

### 例外情况

以下修改可跳过人类闭环，直接进入提交前验证：

- 纯逻辑修复（不触及视觉表现）
- 仅修改测试文件
- 仅修改文档/注释
- 仅修改构建配置

> Agent 应自行判断并声明是否跳过人类验证，但最终以人类确认为准。

## 提交前验证（commit 前必做）

```bash
# 1. 类型检查
npm run typecheck

# 2. Lint 检查（必须零错误）
npm run lint

# 3. 全量测试
npm test

# 4. 库构建
npm run build

# 5. 示例构建（如修改了示例相关）
npm run example:build

# 6. 产物校验（可选，CI 会再做）
ls dist/ui-frame.js dist/ui-frame.umd.cjs dist/style.css dist/index.d.ts
```

**提交信息格式**：

```
<type>(<scope>): <subject>

<body>
```

- `type`: feat/fix/docs/style/refactor/perf/test/build/ci/chore/revert
- `subject`: 不超过 100 字符，使用现在时祈使语气
- 例：`feat(Button): 新增 loading 状态支持`

## 特殊场景处理

### 新增组件

```
1. 创建 composable（逻辑层）
2. 创建 composable.test.ts（测试）
3. 创建 Vue 组件（视觉层）
4. 导出到 src/index.ts
5. 更新示例站点展示
6. 运行完整 pipeline
```

### 修改主题/样式

```
1. 确认 CSS 变量命名符合 Token 规范
2. 验证亮色/暗色双主题
3. 检查 prefers-reduced-motion 支持
4. 确认无障碍属性（ARIA、焦点管理）
```

### 破坏性变更

```
1. 评估影响范围
2. 必要时添加 .changeset 文件
3. 更新文档和示例
4. 在提交信息中明确标注
```

┌─────────────────────────────────────────────────────────────┐
│ 人类闭环验证（UI 修改时）                                   │

│  ├─ 不通过 → Agent 接收反馈重新修改                         │

                          ↓

│  ├─ typecheck（全量）                                       │

│  ├─ npm test（全量）                                        │
│  ├─ npm run build                                          │
│  ├─ npm run example:build（如相关）                         │
