# 文档分层与撰写规范

> 本文档说明 @echolab/ui-frame 项目的文档组织结构，以及 Agent 在撰写或修改文档时应遵循的原则和规范。

---

## 一、文档目录结构

项目文档按读者对象和用途分为三层，分别存放于 `docs/` 下的不同目录：

```
docs/
├── before-develop/          # 开发前必读（约束与规范）
│   ├── design-philosophy.md    # 设计理念
│   ├── develop-pipeline.md     # 开发验证流程
│   ├── documentation-guide.md  # 文档分层与撰写规范
│   ├── project.md              # 索引与快速参考
│   └── readme-specification.md # 项目 README 内容规范
│
├── develop/                 # 开发参考（技术细节）
│   ├── api.md                  # API 参考文档
│   ├── ci-design.md            # CI/CD 设计说明
│   └── guide.md                # 用户指南
│
└── [其他专题文档]/            # 视需要创建
```

---

## 二、各层文档的定位

### `docs/before-develop/` — 约束层

**读者**：参与本项目开发的 Agent 或开发者

**目的**：在编码之前建立共识，回答"该怎么写"的问题

**内容特征**：

- 不写具体 API，只写设计原则和约束规则
- 使用祈使语气，条目清晰可检查
- 包含流程图、检查清单等可执行的结构
- 变更频率低，是项目的"宪法"

**现有文档职责**：

- `design-philosophy.md` — 六条核心设计原则，所有组件实现的判断依据
- `develop-pipeline.md` — 从编码到提交到 CI 的完整验证流程
- `documentation-guide.md` — 文档本身的组织结构与撰写规范
- `readme-specification.md` — 项目根目录 README 的内容结构与撰写标准

---

### `docs/develop/` — 参考层

**读者**：使用本库构建应用的外部开发者

**目的**：回答"怎么用"和"有什么"的问题

**内容特征**：

- 按读者对象拆分为 API 文档和用户指南两份
- 所有可被外部使用的模块必须有文档覆盖
- 类型定义完整，示例代码可直接复制运行

**文档分工**：

| 文档           | 面向读者                 | 内容风格                   | 查询方式       |
| -------------- | ------------------------ | -------------------------- | -------------- |
| `api.md`       | 需要精确接口信息的开发者 | 参考手册式，表格化、类型化 | 按模块名检索   |
| `guide.md`     | 需要学习如何使用的开发者 | 教程式，场景驱动           | 按使用场景阅读 |
| `ci-design.md` | 需要修改 CI/CD 的开发者  | 架构说明式，流程图 + 表格  | 按工作流检索   |

**拆分原则**：

- API 文档聚焦"这是什么" — 精确描述每个导出模块的签名、参数、返回值、类型
- 用户指南聚焦"怎么做" — 以使用场景组织，提供完整的可运行示例
- 两者内容可以交叉引用，但不应重复详述同一内容

## 三、文档撰写规范

### 3.1 通用格式

- 使用 Markdown 格式，一级标题为文档名，二级标题用 `##`，三级用 `###`
- 同级标题之间用 `---` 分隔
- 代码块标注语言（`ts`、`vue`、`scss`、`bash`）
- 中英文混排时，英文与中文之间留一个空格

### 3.2 API 文档撰写规范

**模块组织**：按功能领域分组（基础输入、表单、数据展示、导航、反馈、布局、扩展）

**每个组件/函数的条目必须包含**：

1. **导入语句** — `import { Xxx } from '@echolab/ui-frame'`
2. **类型导入**（如有）— `import type { XxxProps } from '@echolab/ui-frame'`
3. **Props / 参数表格** — 列：名称、类型、默认值、说明
4. **事件表格**（如有）— 列：事件名、参数、说明
5. **插槽表格**（如有）— 列：名称、说明、作用域参数
6. **类型定义**（如有）— 使用 TypeScript interface 展示数据结构
7. **简短示例** — 展示最常见用法，不超过 10 行

**禁止**：

- 不堆砌组件描述性文字，用表格代替大段 prose
- 不重复已在源码类型中表达的信息
- 不写内部实现细节（如 composable 内部状态名）

### 3.3 用户指南撰写规范

**内容组织**：按使用场景而非按模块名组织

**每个场景必须包含**：

1. **场景说明** — 一句话说明解决什么问题
2. **完整可运行示例** — 包含 `<script setup>`、`<template>`、`<style>`
3. **关键代码注释** — 标注需要注意的点
4. **与相关场景的交叉引用** — 如"参见 [表单验证](#表单验证)"

**禁止**：

- 不罗列 API（这是 API 文档的职责）
- 不提供不可运行的代码片段
- 不深入源码实现原理

### 3.4 CI 文档撰写规范

**文件位置**：`docs/develop/ci-design.md`

**读者**：需要理解或修改 GitHub Actions CI/CD 的开发者

**内容组织**：按工作流逐一说明，每个工作流必须包含：

1. **触发条件** — `on:` 的完整配置，包括分支、事件类型、手动触发
2. **任务流水线** — 按执行顺序列出每个步骤的命令和说明
3. **设计要点** — 并发控制、权限配置、产物复用策略、环境变量说明
4. **故障排查** — 常见失败场景的症状、根因与修复方法

**格式要求**：

- 使用表格展示步骤列表（步骤名 | 命令 | 说明）
- 使用 ASCII/文本流程图展示工作流之间的触发关系
- 环境变量和 Secrets 必须说明用途，不暴露具体值
- 产物校验清单必须逐项列出，不可省略

**禁止**：

- 不写与 CI 无关的内容（如组件使用教程、API 说明）
- 不假设读者熟悉 GitHub Actions 基础概念，关键配置应简要解释
- 不遗漏 `workflow_run` 等触发依赖关系的说明

### 3.5 文档维护原则

- **新增可被外部使用的模块** → 必须同步更新 API 文档
- **新增使用场景** → 必须同步更新用户指南
- **修改接口签名** → 必须同步更新两份文档
- **新增/修改 CI 工作流** → 必须同步更新 `ci-design.md`
- **修改 CI 环境配置**（Node 版本、npm 版本、环境变量） → 必须同步更新 `ci-design.md`
- **纯内部重构不触及外部接口** → 无需更新文档
